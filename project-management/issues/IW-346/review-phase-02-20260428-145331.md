## Warnings

- **`test/bootstrap.bats:104-122` — PATH munging is fragile on hosts where `scala-cli`, `java`, or `jq` resolve to identical bin directories or to `/usr/bin`/`/bin` (already in PATH).** The reconstruction `PATH="/usr/bin:/bin:$(dirname $(command -v scala-cli)):$(dirname $(command -v java)):$(dirname $(command -v jq))"` will:
  1. Silently emit empty path components (`::`) if any `command -v` lookup fails — empty components mean "current directory" in POSIX PATH, which is a (mild) safety smell.
  2. Fail to hide Mill if a user (or the CI container) has installed `mill` into `/usr/bin` or `/bin`. The test then becomes a tautology — it passes because Mill is still available, not because the bundled jars work in isolation.
  Consider a stricter approach: build a temp `bin/` directory containing only symlinks to the four required binaries, set `PATH` to just that, and `assert ! command -v mill` before exercising the launcher. That way the test's premise is verified, not assumed. As-is, the test catches the "PATH lookup of `mill` returns nothing" failure mode but not the "Mill is reachable but launcher mistakenly invokes it" failure mode the regression actually targets.

- **`test/bootstrap.bats:109,121` — `local saved_path=...` outside a function is a soft bug.** `local` is only valid inside a function body; in BATS each `@test` desugars to a function (so this happens to work in current BATS), but some shells (and older bats-core releases) reject it. Since BATS already scopes `PATH` assignments inside an `@test` block to the test, the manual `saved_path` save/restore at line 109 and the restore at line 121 are unnecessary — drop both lines and the test is simpler and more portable. (See the checklist note in the prompt: "in BATS, PATH= assignments inside @test blocks are local to the test.")

- **`.github/workflows/release.yml:22-26` — `env:` block on the `Run all tests` step is defensive parity with `ci.yml:test` but is currently redundant.** `./iw ./test` on this branch is the BATS+unit suite; nothing in that suite needs the Web Awesome registry today. That's fine — keep the parity — but worth flagging that the safety net only kicks in if the test command grows a Mill-driven dashboard build path. If you intend that as an invariant, add a one-line comment so the next person doesn't drop the env block as "unused."
