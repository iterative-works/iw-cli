## Testing Review

### Critical Issues
- **`ServerClientTest` — `assume(IW_SERVER_DISABLED=1)` silently skips the test** on `./mill dashboard.test` because Mill's test fork does not set the env var. The old reflection-based `withEnv` was correctly removed (Java 9+ module access), but the replacement degrades coverage to zero. Fix: either set `forkEnv` in `build.mill` for `dashboard.test`, or make `ServerClient` accept the disabled flag as a pure parameter for test.
- **`dashboard-dev-gate.bats` — tests 1–2 wrap the Mill build inside a `timeout 30`**, risking false timeouts on cold checkouts; `pkill -f "iw-dev-gate-test"` in teardown matches nothing (process is never named that). Add an explicit "jar pre-built" precondition check and fix the `pkill` pattern (or drop it and use PID-based cleanup).

### Warnings
- **`startTestServerDevMode` duplicates `startTestServer`** (~55 lines). Extract one helper taking `DevModeConfig`.
- **`dashboard-rebuild-gate.bats` test 2 asserts on Mill's stdout** — fragile across Mill versions. Either gate at the shell-script level (stub `mill` on `$PATH`) or treat as manual-only.
- **No test covers the `CaskServer.start` warning path** when `devMode=true && viteEnv.isEmpty`. Add a unit test capturing `System.err`.

### Suggestions
- **`AssetUrlTest`** — add a leading-slash edge case (`"/main.js"`) to pin behavior.
- **`core/test/SampleDataTest.scala`** uses `package iw.tests` while other core tests use `iw.core.test`. Align or document the convention.

</review>

## Security Review

### Critical Issues
None found.

### Warnings
- **`mill_jar_path` silently returns empty on Mill failure** — `iw-run:28-33` has no `set -o pipefail` and no explicit exit-status check. Downstream `ensure_dashboard_jar` would export `IW_DASHBOARD_JAR=""`, producing a confusing `java -jar ""` failure instead of a clean abort.
- **`hosts` argument in `ProcessManager.spawnServerProcess` passes unvalidated comma-joined string** — values recovered by `ServerDaemon.main` via `.split(",")`. Defense-in-depth validation missing (Undertow rejects deep in the daemon log).
- **`AssetUrl` does not validate path/query in `VITE_DEV_URL`** — `DevModeConfig.validate` checks only scheme + host. A crafted path/query in `viteDevUrl` is concatenated with `/src/$path`. Scalatags attribute escaping blocks direct XSS, but tightening the contract to "origin-only URL" makes intent explicit.

### Suggestions
- `SampleDataCli.main` does no path validation — low impact today (subprocess of `iw dashboard`) but worth a `$HOME`-scoped sanity check if the jar is ever broadly distributed.

**Verified:** double-gate non-bypassable, exact-equality loopback set, http-only scheme, non-zero exit on refusal, `ProcessBuilder(Seq[String]*)` safe from shell injection, `IW_DASHBOARD_JAR` treated as a resolved file path.

</review>

## Architecture Review

### Critical Issues
- **`./iw server start` path broken** — `iw-run` only calls `ensure_dashboard_jar` for `actual_name == "dashboard"`. But `commands/server.scala` (and `ServerClient.ensureServerRunning` auto-start) invoke `ProcessManager.spawnServerProcess`, which now reads `IW_DASHBOARD_JAR` and `sys.error`s when absent. Any server-start path raises an uncaught error. Fix: extend the guard to include `server` (and possibly unconditionally via `execute_command`).

### Warnings
- **`"iw.dashboard.SampleDataCli"` hardcoded in `commands/dashboard.scala:113-119`** trades compile-time coupling for stringly-typed runtime coupling. Mark the class name as stable public API in `SampleDataCli.scala`'s PURPOSE header.
- **`CaskServer.start` reads `VITE_DEV_URL` from `sys.env`** — should be read at the `ServerDaemon.main` shell boundary and passed as a parameter to keep env I/O at the outermost edge.

### Suggestions
- `mill_jar_path` sed regex brittle against future Mill ref-format changes — add an existence check on the resolved path.
- `PageLayout` banner check `!= DevModeConfig.Off` → pattern match for exhaustiveness (duplicate of style/scala3 finding).

</review>
