## Security Review

### Critical Issues

#### `javascript:` URL injection via `trackerBaseUrl` in the repo link `href`
**Location:** `core/model/RepoUrlBuilder.scala:24-25`
For GitLab tracker type, `buildRepoUrl` assembles the href directly from `trackerBaseUrl` with no scheme check. A value of `javascript:alert(document.cookie)//` survives Scalatags HTML escaping and ends up as a working `href`. Threat model: a malicious `.iw/config.conf` in a supply-chain-compromised repo reaches the browser unsanitised.

**Recommendation:** Add a scheme allow-list in `RepoUrlBuilder.buildRepoUrl` (filter to `http`/`https`).

#### `repository` field: weak validation allows arbitrary path characters in constructed URL
**Location:** `core/model/Config.scala:245-251`
A `repository` value of `/../../../etc/passwd` passes the existing `nonEmpty segments` check, producing `https://github.com/../../../etc/passwd` â€” open-redirect / path-traversal via crafted config.

**Recommendation:** Character allow-list per segment (`[a-zA-Z0-9._-]+`).

### Warnings

#### `target="_blank"` without `rel="noopener noreferrer"` on the new repo link
**Location:** `dashboard/jvm/src/presentation/views/WorktreeCardRenderer.scala:143`
Tabnabbing surface. Modern browsers default to `noopener` for cross-origin `_blank` but explicit attribute is clearer and adds `noreferrer`.

#### `trackerBaseUrl` accepted without URL scheme validation at parse time
**Location:** `core/model/Config.scala:267-270`
Same root cause as critical #1 â€” also affects tracker issue links throughout `CaskServer.scala`. Validate at parse time.

### Suggestions

#### No protocol-relative or `data:` guard on `PullRequestData.url`
Source is the GitHub/GitLab API (trusted), but a `SafeUrl` value type would document intent. Lower priority.
