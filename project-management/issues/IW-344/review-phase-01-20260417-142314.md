The new code introduces `IW_CORE_JAR` as a user-overridable path. The value is used in three places:

- `mkdir -p "$(dirname "$CORE_JAR")"` — quoted, safe.
- `-o "$CORE_JAR"` passed to `scala-cli` — quoted, safe.
- `-newer "$CORE_JAR"` passed to `find` — quoted, safe.

There is no `eval`, no `source` of user-controlled paths, and no unsanitized command substitution introduced by this diff.

### Warnings

#### `IW_CORE_JAR` accepts an arbitrary writable path without validation
**Location:** `iw-run:14-15`
**Problem:** `CORE_JAR="${IW_CORE_JAR:-$INSTALL_DIR/build/iw-core.jar}"` accepts any path the caller supplies. The risk is low in the tool's intended use (developer workstation) but higher in automated pipelines where env vars may be injected by untrusted build inputs.
**Recommendation:** If the variable is intended only for developer overrides, document that expectation. If it may be set in CI, add a basic path sanity check.

### Suggestions

#### `core_jar_stale` silently swallows `find` errors
**Location:** `iw-run:55-61`
**Problem:** `2>/dev/null` suppresses all `find` errors, including permission-denied on the core directory. Could mask a misconfigured `CORE_DIR` by appearing to succeed.
**Recommendation:** Log a warning on non-zero exit, or narrow the suppression.

#### `IW_CORE_JAR` is exported into child process environment
**Location:** `iw-run:15`
**Problem:** Exports the resolved (potentially user-supplied) path to every subprocess.
**Recommendation:** Likely intentional (Phase 2 commands will need it). No change needed.
