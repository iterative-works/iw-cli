## Warnings

- **`ensure_core_jar` now runs before command-name validation and existence checks** (`iw-run:519`). In the prior code, the `core_files=$(find ...)` collection happened *after* each branch validated the command name regex and confirmed the command file existed. After Phase 2, `./iw ./bad!name` or `./iw nonexistent` triggers a `core_jar_stale` find (and, if stale, a full `scala-cli package` rebuild) before emitting the "Invalid" / "not found" error. Minor behavioral regression; the rebuild cost is only paid when the jar is already stale, so impact is bounded. Acceptable per the phase-02-context.md contract ("freshness is a property of `execute_command` itself"), but worth documenting because it changes the error-path timing.

- **`# shellcheck disable=SC2086` on the project-command `exec` line is now stale** (`iw-run:550`). The project branch no longer interpolates any word-split variable on that line — all arguments are quoted or literal. SC2086 would not fire here, so the directive is dead. The plugin and shared branches still need it (they word-split `$hook_files` / `$lib_files`). Either drop the directive on line 550, or leave a one-line comment explaining it is retained defensively. The context doc (section "Argument-order rationale", final bullet) actually mandates keeping it, but that rationale is mechanical, not technical — worth reconciling.

## Suggestions

- **`-f` on `scala-cli package`** (`iw-run:44`): correct fix for the Phase 1 latent bug; explicitly in scope per the session agreement. No objection.

- **Argument-order consistency across branches is good.** All three `exec` lines now follow `cmd_file → branch-sources → project.scala → --jar CORE_JAR → -- "$@"`. This makes the three invocations structurally identical modulo the source-file list and matches the spec in phase-02-context.md section "scala-cli run invocation pattern".

- **`IW_HOOK_CLASSES` propagation preserved.** The plugin branch (`iw-run:630`) and shared branch (`iw-run:710`) both retain `IW_HOOK_CLASSES="$hook_classes" exec scala-cli run ...`. Exit code, stderr/stdout ordering, and argv ordering after `--` are unchanged by the diff — only the classpath-providing flags differ.

- **Concurrent-rebuild race is explicitly accepted** in phase-02-context.md ("Risk: concurrent invocations..."). `scala-cli package` with `--server=false` does a close-to-atomic write-and-rename, and the worst case is a duplicated 1.1s rebuild. No lock file needed for this phase. Flag for a later hardening pass if the tool moves into multi-agent / CI-parallel territory.

- **No new injection surface.** `$CORE_JAR` and `$CORE_DIR` are both quoted everywhere they appear in the new `exec` lines; the only word-split vars are the pre-existing `$hook_files` / `$lib_files`, which are built from `find` output and constrained by the hook-filename regex. No user input flows into the new interpolations.

- **Deferred BATS coverage is acceptable per phase-02-tasks.md line 15.** Verification was done via the manual smoke-test matrix (tasks.md lines 32-39, all checked except the plugin-command case which is N/A in this repo). Phase 3 BATS work should include: missing-jar auto-rebuild, stale-jar auto-rebuild, `IW_CORE_JAR` override, and a regression test on `test/project-commands-execute.bats` to catch jar-mode classpath-visibility breakage.

- **`core/project.scala` is passed as both a source file and is also compiled-into the jar via `build_core_jar`'s find-glob.** This duplication is intentional and explained in phase-02-context.md ("Note: `project.scala` appears twice..."): scala-cli only honors `//> using dep` directives from source-set files, not from jar-packaged sources, so the file must be re-passed at run time. Worth a one-line code comment at the `exec` sites so future readers don't think it's redundant.

- **Comment above the shared-branch `exec` still says "including core library and any discovered hook files"** (`iw-run:707-708`). The phrasing is still accurate (the jar *is* the core library), but readers diffing this commit may be briefly confused by the change in how "core library" is delivered. Low priority; optional tightening to "...including core jar and any discovered hook files".
